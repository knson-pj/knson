## 3. 미해결 사항 — 두 가지

### 3-1. 화면 일괄 실행 박스의 `inserted=0` 항상 표시 (CRITICAL — 사용자 불신 원인)

- **증상**: 화면 일괄 실행 결과 박스에 `processed=50 inserted=0` 으로 항상 표시. 실제 DB 에는 박히고 있는데도 사용자는 "안 됐다" 로 인지.
- **원인**: 백엔드 응답 필드가 `flrInserted` / `jjInserted` / `wclfInserted` (extras) 와 `inserted` (price) 로 모드마다 다른데, 화면이 단일 `inserted` 필드만 합산함. extras 응답에서는 `inserted` 가 undefined → 0 표시.
- **해결 위치**: 화면 컴포넌트 코드 (Edge Function 아님). 사용자 측 프론트엔드 저장소.
- **권장 처리**:
  - 옵션 A (화면 측 수정): 일괄 실행 응답 처리에서 모드별로 `flrInserted + jjInserted + wclfInserted` 합산
  - 옵션 B (백엔드 호환 필드 추가): `handleEnrichExtras` 응답에 `inserted = flrInserted + jjInserted + wclfInserted` 누적 필드 추가 (작업 규칙 3번 — 인플레이스 누적). 이게 사용자 입장에서 가장 빠른 해결.

**옵션 B 가 강력 권장**. 백엔드 한 줄 추가로 화면 코드 안 건드리고 끝남.

### 3-2. 진척 속도가 만족스럽지 못함 (조사 필요)

- **수치**: 1회 배치 `processed=50` → 층별 칩 10 → 39 (29건물 진척). 비율 58%.
- **가능한 정상 사유**:
  - 일부 형제가 부속건축물이라 floor 응답에 mgm_bldrgst_pk 가 없음 (분배 매칭 실패 → 자기 PK 미박힘 → 다음 라운드 재조회) → 영원히 매 라운드 sibling skip 후보로 잡힘
- **조사 SQL**:
  ```sql
  -- 역삼동 표제부 보유 중 floor 박힌 건물 (parcel 형제 검사 없이)
  SELECT
    count(DISTINCT b.id)                                     AS total_title_buildings,
    count(DISTINCT b.id) FILTER (WHERE f.id IS NOT NULL)     AS with_floor,
    count(DISTINCT b.id) FILTER (WHERE f.id IS NULL)         AS without_floor
  FROM public.buildings b
  LEFT JOIN public.building_floor_outline f ON f.building_id = b.id
  WHERE b.sigungu_cd='11680' AND b.bjdong_cd='10300'
    AND b.mgm_bldrgst_pk IS NOT NULL;

  -- 같은 번지에 floor 박힌 형제가 있지만 자기는 안 박힌 row 수 (정상 케이스 — 부속건축물 등)
  WITH parcel_floor AS (
    SELECT b.sigungu_cd, b.bjdong_cd, b.bun, b.ji,
           bool_or(f.id IS NOT NULL) AS any_sib_has_floor
    FROM public.buildings b
    LEFT JOIN public.building_floor_outline f ON f.building_id = b.id
    WHERE b.sigungu_cd='11680' AND b.bjdong_cd='10300'
    GROUP BY b.sigungu_cd, b.bjdong_cd, b.bun, b.ji
  )
  SELECT count(*) AS sibling_already_done_but_self_missing
  FROM public.buildings b
  LEFT JOIN public.building_floor_outline f ON f.building_id = b.id
  JOIN parcel_floor pf USING (sigungu_cd, bjdong_cd, bun, ji)
  WHERE b.sigungu_cd='11680' AND b.bjdong_cd='10300'
    AND b.mgm_bldrgst_pk IS NOT NULL
    AND f.id IS NULL
    AND pf.any_sib_has_floor = true;
  ```
- **해석**:
  - `sibling_already_done_but_self_missing` 가 양수면 v6.4.5#2 필터가 그 번지를 "처리됨" 으로 정확히 판정해 skip 함 (정상 동작 — 그 row 는 영원히 박히지 못함, 정부 API 응답에 없으니까)
  - 0 이면 진척 결함이 다른 곳에 있음 — 추가 조사 필요

## 6. v6.4.5 코드의 핵심 설계 결정

### 6-1. `handleEnrichExtras` (floor / jijigu / wclf)

1. **rows 쿼리**: 동의 모든 표제부 보유 row 를 limit=10000 으로 한 번에 받음 (4,957 < 10,000 안전 범위)
2. **처리 상태 일괄 조회**: `building_id` 청크(500개씩) 로 floor / jijigu / wclf 각 모드 박힌 PK 를 일괄 조회 → 3 × ceil(N/500) PostgREST 호출
3. **번지 단위 그룹화**: candidates 를 `(bun, ji)` 키로 묶어 parcelMap 생성
4. **미처리 번지 선별**: 세 모드 모두 박힌 parcel 은 skip, 나머지 중 limit 개의 firstRow 만 처리 사이클로 진입
5. **처리 사이클** (v6.4.4 부터 유지):
   - 번지 단위로 sibling 조회 후 정부 API 1회 호출
   - 응답의 `mgm_bldrgst_pk` 로 형제별 분배
   - INSERT 안전망: duplicate key 발생 시 warning + skip (errors++ 안 함)
6. **fetch 함수 dedup** (v6.4.5#1):
   - floor: `(mgm_bldrgst_pk, flr_gb_cd, flr_no)` 키 dedup
   - jijigu: `(mgm_bldrgst_pk, jijigu_gb_cd, jijigu_cd)` 키 dedup
   - hsprc: `(mgm_bldrgst_pk, std_day)` 키 dedup
   - wclf: dedup 안 함 (한 건물 1줄 fallback 으로 처리)

### 6-2. `handleEnrichPrice` (hsprc)

- 동일 패턴이나 처리 상태 조회는 `parent_mgm_bldrgst_pk in.(...)` 청크 (FK 없이 텍스트 매칭)
- 분배 키: `building_units` 의 `(mgm_bldrgst_pk → parent_mgm_bldrgst_pk)` 매핑 활용 (v6.4.2 의 정확한 호실 분배 결과 신뢰)

## 7. 다음 작업 — 권장 순서

### STEP 1. 백엔드 응답 호환 필드 추가 (HOTFIX v6.4.5#3 또는 v6.4.6)

`handleEnrichExtras` 의 두 응답 객체 (정상 + rateLimited) 에 `inserted` 누적 필드 추가:

```ts
// 정상 응답
return J({
  ok:true, dongCode:dc, processed,
  flrInserted, jjInserted, wclfInserted,
  inserted: flrInserted + jjInserted + wclfInserted,  // ← 추가
  errors
});

// rateLimited 응답
return J({
  ok:true, dongCode:dc, processed,
  flrInserted, jjInserted, wclfInserted,
  inserted: flrInserted + jjInserted + wclfInserted,  // ← 추가
  errors,
  rateLimited: true,
  note: "..."
});
```

이 한 줄 패치 후 사용자에게 단일 `index.ts` 전달 → 배포 → 화면 `inserted=N` 정상 표시 확인.

### STEP 2. 진척 속도 의문 조사

위 §3-2 의 두 SQL 을 사용자에게 실행 요청. 결과로 정상/비정상 갈림:
- `sibling_already_done_but_self_missing` > 0 → v6.4.5#2 가 정확히 skip 중 (정상)
- = 0 → 다른 결함 → 로그 재조사

### STEP 3. 역삼동 전체 검증

화면 `inserted` 표시 정상화된 후, 역삼동 일괄 실행을 자동 반복으로 끝까지 돌려서:
