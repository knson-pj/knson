### 2-4. v6.4.6#1 — sentinel 핫픽스 (CRITICAL)

v6.4.6 배포 후 실제 검증에서 두 결함 발견:

**결함 ①** — 처리 사이클의 모드별 skip 판정이 DB 박힘만 보고 sentinel 안 봄. 메모리 필터가 "한 모드라도 미처리면 후보 진입" 정책이라, 어떤 번지가 floor 만 미처리이고 jj/wclf 는 sentinel 박혀 있어도 후보로 들어왔음. 처리 사이클에서 jj/wclf 정부 API 또 호출 → 또 빈 응답 → 같은 sentinel INSERT 두 번째 시도 → UNIQUE 위반 dup 에러.

**결함 ②** — `markSkipParcel` 의 PostgREST URL 에 `on_conflict` 누락. `Prefer: resolution=merge-duplicates` 만 있으면 PostgREST 가 PK(id, autoincrement) 만 conflict 대상으로 봐서 일반 INSERT 가 됨. UNIQUE 인덱스(uq_skip_parcels) 위반 시 23505 에러로 throw.

**수정**:
1. 처리 사이클의 4곳 skip 판정에 sentinel set 조회 추가 (floor/jijigu/wclf/hsprc)
2. `markSkipParcel` URL 에 `?on_conflict=sigungu_cd,bjdong_cd,bun,ji,mode` + `Prefer: ignore-duplicates`

**검증 결과** — 같은 역삼동 재실행:
| 모드 | v6.4.6 추가 호출 | v6.4.6#1 추가 호출 | 감소율 |
|---|---|---|---|
| floor | +110 | +2 | 98% ↓ |
| jijigu | +113 | 0 | 100% ↓ |
| wclf | +229 | 0 | 100% ↓ |

**floor +2** 는 정상 — 이전 실행에서 jj/wclf 만 sentinel 박혔던 번지의 floor 미처리분이 이번에 채워진 것 (의도된 동작).

## 4. v6.4.6#1 코드 핵심 설계 정리

### 4-1. sentinel 흐름

1. **후보 선별 단계** (handleEnrichExtras / handleEnrichPrice)
   - candidates: 동의 표제부 보유 row 전체 (limit=10000)
   - 박힌 PK 청크 조회 (floor/jijigu/wclf 또는 hsprc parent)
   - **sentinel 동 단위 한 번 조회** (limit=20000) → set 적재
   - 번지 단위 그룹화 후 done 판정: `(박혔거나) OR (sentinel)` 인 모드는 done

2. **처리 사이클** (한 번지당)
   - 각 모드 skip 판정도 `(DB 박힘) OR (sentinel)` ← v6.4.6#1 핵심
   - skip 인 모드는 정부 API 호출 자체 안 함
   - 호출한 모드가 빈 응답이면 `markSkipParcel` 으로 sentinel upsert

3. **markSkipParcel (헬퍼)**
   - URL: `/rest/v1/building_collect_skip_parcels?on_conflict=sigungu_cd,bjdong_cd,bun,ji,mode`
   - Header: `Prefer: return=minimal,resolution=ignore-duplicates`
   - INSERT 실패해도 처리는 계속 (warn log 만)

### 4-2. 효과 검증된 부분

- 빈 응답 번지가 다음 라운드부터 자동 skip
- 같은 동 재실행 시 정부 API 호출 거의 0 (이미 처리 + sentinel 모두 skip)
- dup warn 로그 사라짐
- floor 정상 진척만 추가 호출 발생

### 4-3. 의도된 한계 (자연스러운 양상)

- 분모 4,957 의 100% floor 도달은 정부 API 응답 분포상 불가능 — 데이터 있는 번지만 박힘
- 부속건축물 형제는 응답에 없는 게 정상 — 그 형제는 자연 누락
- 메모리 필터 "어느 형제든 박히면 번지 done" 정책이 그 누락을 자연 처리

## 10. 트러블슈팅 — 본 대화에서 학습된 패턴

### 정부 API 한도 의문 시 우선 확인 순서
1. 사용량 카드 캡쳐로 모드별 호출수 비교
2. `building_collect_skip_parcels` 누적 수 확인
3. Logs 탭에서 `sentinel` 검색 → dup 로그 있으면 핫픽스 필요

### sentinel 작동 검증 SQL
```sql
-- 모드별 누적
SELECT mode, count(*), max(created_at)
FROM public.building_collect_skip_parcels
WHERE sigungu_cd='11680' AND bjdong_cd='10100'
GROUP BY mode ORDER BY mode;

-- 전체 분포
SELECT sigungu_cd, bjdong_cd, mode, count(*)
FROM public.building_collect_skip_parcels
GROUP BY sigungu_cd, bjdong_cd, mode
ORDER BY count(*) DESC LIMIT 20;
```

### 화면 캡쳐 분석 우선순위
- 진행 박스의 `processed=N inserted=M` 비교
- 사용량 카드의 모드별 호출수 변화
