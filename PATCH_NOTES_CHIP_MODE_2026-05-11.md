# 진행 칩 표시 방식 모드별 차별화 (2026-05-11)

## 문제
사용자 지적: "표제부가 4,957이라고 해서 전유+공용 등의 분모가 4,957이라는 건 오류이다"

**맞는 지적**. 모드마다 대상이 다른데 모두 동 전체 건물(표제부 수)을 분모로 쓰던 코드.
- 전유+공용 → 집합건축물만 대상인데 분모가 전체 건물 → 영원히 빨강 (잘못된 시그널)
- 부속지번 → 합필된 건물에만 데이터 있음
- 층별등 → 데이터 등록 여부에 따름
- 등등

DB 확인 결과 역삼동 집합건물 1,401건 중 114건이 정상 처리되고 1,287건이 sentinel(빈 응답)로
모두 처리된 상태인데도, 화면은 "전유+공용 114/4,957"로 표시되어 "수집 안 됨"으로 오해됨.

## 수정

`admin-tab-buildings.js` 의 `renderStatus` 함수 안의 진행 칩 호출부:

| 모드 | 변경 전 | 변경 후 |
|---|---|---|
| 표제부 | 분수 (titleCnt / totalBld) | 그대로 (의미 명확) |
| **전유+공용** | 분수 (v2Cnt / titleCnt) ❌ | **binary "N건"** |
| 지오코딩 | 분수 (geoCnt / titleCnt) | 분수 (geoCnt / totalBld) — 분모만 정확화 |
| **부속지번** | 분수 (atchCnt / titleCnt) ❌ | **binary "N건"** |
| 총괄표제 | binary | 그대로 |
| **층별등** | 분수 (extrasCnt / titleCnt) ❌ | **binary "N건"** |
| 공시가 | binary | 그대로 |

추가: `pchip()` 함수에 4번째 인자 `description` 추가 → 호버 시 모드별 의미 툴팁 표시.
0건이 "안 됨"인지 "원래 없음(예: 단지 정보 없는 동)"인지 사용자가 헷갈리지 않게.

호버 툴팁 예시:
- 전유+공용 → "전유부 호실 데이터가 저장된 건물 수 (집합건축물만 해당)"
- 부속지번 → "부속 지번 정보가 있는 건물 수 (합필된 건물만 해당)"
- 총괄표제 → "단지 단위 총괄 정보가 있는 건물 수 (대단지만 해당)"
- 공시가 → "공시가격 정보가 있는 호실/건물 수 (집합·거주만 해당)"

## 변경 통계
- `admin-tab-buildings.js`: +12 줄 / -9 줄 (pchip 함수 + 호출부)
- 다른 파일: 변경 없음

## 배포
운영 레포의 `admin-tab-buildings.js` 한 파일만 교체.
캐시 무효화: `?v=20260511-ux5` 등으로 갱신 권장.

## 추후 진행 권장 (이번 범위 밖, 다음 패치에서 다룸)

### 1. 정부 API 한도 초과 sentinel 오염 검사·복구
현재 의심점: 오늘 사용량 11,287/10,000(115%) 초과 후 받은 빈 응답이
1,287건의 sentinel(-1) 처리에 일부 포함됐을 수 있음.
이 건물들은 내일 한도 리셋 후에도 다시 안 잡힘 (`unit_total_area=eq.0` 필터에서 제외).

**필요한 작업:**
- (Edge Function) 정부 API 응답의 `resultCode === "22"` (한도 초과) 감지 → sentinel 처리 안 함
- (DB) sentinel(-1) 행 중 한도 초과 시점에 처리된 행을 0으로 되돌리는 정리 SQL

### 2. 모드별 정확한 분모 표시 (옵션)
Edge Function 응답에 모드별 모집단 수를 함께 반환하면 binary 칩에도 "114건 / 1,401 집합건물 중" 같은
부가 정보 표시 가능. 다만 응답 한 번 더 무거워지므로 이번엔 보류.
